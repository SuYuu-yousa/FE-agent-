# 基本概念

- **Next.js** 是基于 **React** 的框架

- **Nuxt.js** 是基于 **Vue** 的框架

- 它们常被叫做 **全栈框架**

- 它们支持 **SSR / SSG / CSR** 等页面渲染方式

- **确实会让 React/Vue 的一部分代码在服务器运行**

- 但不是简单地“把整个前端搬到服务器”，而是：

  - 有些代码在服务器跑

  - 有些代码在浏览器跑

  - 两边协作完成网页


------

## 什么叫“全栈框架”？

> **全栈框架 = 不只管前端页面，还顺带帮你处理服务端能力的框架**

普通 React / Vue 更多像是“写界面”的工具。
而 Next / Nuxt 不只是写界面，还会帮你处理：

- 路由
- 页面渲染
- 服务端渲染
- 静态生成
- 接口/API
- 部署优化
- SEO 相关能力

所以它不是单纯前端框架，而是“前端 + 一部分后端能力”都给你打包好了。

------

# 2. React / Vue 本身有什么局限？

比如你只学 React，自己写一个项目，默认情况通常是：

1. 浏览器先拿到一个很空的 HTML
2. 再下载 JS
3. React 在浏览器里执行
4. 然后把页面渲染出来

这叫 **CSR（Client-Side Rendering，客户端渲染）**

这有几个问题：

- 首屏可能慢
- 搜索引擎有时不容易直接拿到完整内容
- 路由、构建、接口这些东西你要自己搭很多

所以 Next/Nuxt 出现了：
它们把这些常见需求整合好了。

------

# 3. SSR 是什么？

**SSR = Server-Side Rendering（服务端渲染）**

> 页面 HTML 先在服务器生成好，再发给浏览器

------

## 不用 SSR 时

假设你访问一个 React 普通项目：

- 服务器返回一个空壳 HTML，然后浏览器下载 JS，JS 执行后，页面内容才出现

所以你“最开始看到的页面”其实不是服务器直接给的完整页面。

------

## 用 SSR 时

如果是 Next / Nuxt 的 SSR 模式：

- 用户请求页面
- 服务器先运行 React/Vue 代码
- 生成完整 HTML
- 再把这个 HTML 发给浏览器
- 浏览器先立刻看到内容
- 然后 JS 再接管页面交互

这个过程叫：

- **服务端先渲染**
- 浏览器再“接管”交互（接管过程常叫 hydration，水合）

------

## SSR 的好处

1）首屏更快：用户一打开就能先看到内容，而不是等 JS 都下载完。

2）SEO 更好：搜索引擎爬虫更容易直接拿到完整 HTML 内容。

3）更适合内容型网站：博客、新闻

- 官网
- 电商商品详情页

------

## SSR 的代价

也不是没有成本：

- 服务器压力更大
- 每次请求都可能要现算 HTML
- 开发时要注意哪些代码能在服务器运行，哪些只能在浏览器运行

比如：

```
window.localStorage
document.getElementById()
```

这些浏览器对象在服务器上是没有的。

------

# 4. SSG 是什么？

**SSG = Static Site Generation（静态站点生成）**

意思是：

> 页面在“构建时”就提前生成好 HTML 文件，不用等用户来访问时再生成

------

## SSR 和 SSG 的区别

### SSR

- 用户每请求一次
- 服务器就生成一次 HTML

### SSG

- 项目部署/build 的时候
- 就把 HTML 提前生成好了
- 用户访问时直接返回现成文件

------

## 举例

假设你有一个博客网站：

### 用 SSR

每次有人访问 `/post/1`
服务器实时查数据、生成 HTML、返回

### 用 SSG

你部署项目时就把 `/post/1` 的 HTML 生成好
用户访问时直接返回这个静态 HTML

------

## SSG 的好处

- 非常快
- 对服务器压力小
- 部署简单
- SEO 也好

------

## SSG 的缺点

- 不适合“每秒都在变”的页面
- 如果内容更新了，可能需要重新构建

------

# 5. CSR 又是什么？

刚才提过，完整说一下：

**CSR = Client-Side Rendering（客户端渲染）**

就是：

> 页面主要在浏览器里靠 JS 渲染出来

这也是很多传统 React 单页应用的默认方式。

------

## CSR 流程

1. 浏览器请求页面
2. 服务器返回基础 HTML
3. 浏览器下载 JS
4. React/Vue 在浏览器执行
5. 页面内容出现

------

## CSR 的特点

优点：

- 前后端分离清晰
- 交互体验好
- 适合后台管理系统、登录后应用

缺点：

- 首屏可能慢
- SEO 一般不如 SSR/SSG

------

# 6. 所以 Next / Nuxt 到底做了什么？

你可以理解成：

> 它们给 React/Vue 提供了“多种渲染方式 + 服务端能力 + 工程化能力”

它们并不是只做 SSR，实际上它们帮你处理了：

- **页面路由**
- **服务端渲染 SSR**
- **静态生成 SSG**
- **客户端渲染 CSR**
- **接口/API**
- **打包构建**
- **性能优化**
- **SEO 支持**

所以你开发网站时，不用从零拼各种工具。

------

# 7. “让 React 和 Vue 在服务器上运行”这句话对吗？

**对，但不完整。**

更准确说法是：

> Next / Nuxt 可以让 React / Vue 组件在服务器端先执行一次，生成 HTML，再把结果发给浏览器。

注意重点不是“永远在服务器跑”，而是：

- **首轮渲染** 可以在服务器
- **后续交互** 主要还是在浏览器

------

## 举个流程例子

你访问一个商品页：

### 使用 Next SSR 时

1. 浏览器请求 `/product/123`
2. Next 服务器拿到请求
3. 去数据库/接口取商品数据
4. 用 React 组件生成 HTML
5. 把 HTML 发给浏览器
6. 浏览器先显示商品信息
7. JS 加载后，按钮点击、加入购物车等交互开始生效

所以是：

- **HTML 先由服务器生成**
- **交互后由浏览器接管**

------

# 8. 为什么叫“同构 / 通用”思想？

你以后可能会听到两个词：

- **同构**
- **isomorphic**
- **universal rendering**

大意是：

> 同一套组件代码，既能在服务器运行，也能在浏览器运行

比如同一个 React 组件：

```
export default function Page() {
  return <h1>Hello</h1>
}
```

这段组件代码：

- 可以在服务器执行，生成 `<h1>Hello</h1>`
- 也可以在浏览器执行，绑定事件、完成交互

这就是 Next/Nuxt 很核心的思想。

------

# 9. 什么时候用 SSR？什么时候用 SSG？什么时候用 CSR？

| 渲染方式                          | 适合场景                         | 典型页面                                         | 主要原因                                                     |
| --------------------------------- | -------------------------------- | ------------------------------------------------ | ------------------------------------------------------------ |
| **SSR**（Server-Side Rendering）  | 内容经常变化，且需要 SEO         | 电商商品详情、新闻网站、搜索结果页、个性化首页   | 首屏加载快；搜索引擎能直接看到内容；数据更新频繁，适合按请求实时生成 |
| **SSG**（Static Site Generation） | 内容更新不频繁，可提前生成       | 博客文章、官方文档、公司官网、落地页             | 页面可以在构建时生成；访问速度快；服务器压力小；不需要每次请求都实时渲染 |
| **CSR**（Client-Side Rendering）  | 更偏应用型，SEO 不重要，交互复杂 | 后台管理系统、数据看板、内部系统、登录后用户中心 | 页面交互复杂；更适合在浏览器端处理状态和逻辑；通常不依赖搜索引擎收录 |
