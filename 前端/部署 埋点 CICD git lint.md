# git

## rebase和merge

- `merge` 操作不会修改已有的提交，只是创建一个新的合并提交来连接两个分支的历史。

Merge 能永久保留原始提交 ID，便于追溯；Rebase 会生成新 ID，原始 ID 在历史中不可见，仅能通过本地 reflog 临时找回。

- `rebase` 会将当前分支的提交“移植”到目标分支的最新提交上，这一过程会重新生成新的提交对象。
- 新提交的内容可能相同，但哈希值（提交 ID）会改变，因为父提交、时间戳等都变了。
- 原来的提交 ID 在常规的 `git log` 历史中不再出现，仿佛从未存在过

## husky 

作用：

- 在你 `commit`、`push` 前自动执行脚本、
- **husky 是本地检查，`.gitlab-ci.yml` 这类是远程流水线配置。**

常见用途：

- 提交前跑 `lint`
- 提交前跑测试
- 检查 commit message

比如：

- `pre-commit`：提交前执行
- `commit-msg`：检查提交信息格式

一句话：
**husky = 帮你把一些检查挂到 Git 提交流程里。**



# Lint

###### 通常装在项目里，作为 **项目级依赖**，不是“本地编辑器插件”本身。



#### 常见工作流

- 保存时自动修复
- 提交前检查（husky + lint-staged）
- CI 中执行 lint，防止不规范代码进入主分支
